## 1. RAG란 무엇인가

### 한 줄 요약

> **RAG = 검색(Retrieval) + 생성(Generation)**
> "LLM에게 질문하기 전에, 관련 문서를 먼저 찾아서 함께 넘겨주는 것"

### 왜 필요한가

LLM(GPT 등)은 학습 데이터에 없는 것을 모릅니다.
- 사내 테이블 구조? 모릅니다.
- 우리 회사 수율 분석 프로세스? 모릅니다.
- 특정 설비의 챔버 정보? 모릅니다.

RAG는 이 문제를 해결합니다:

```
[기존 방식]
사용자 질문 → LLM → 답변 (학습 데이터에서만)

[RAG 방식]
사용자 질문 → 관련 문서 검색 → LLM + 검색된 문서 → 답변 (우리 데이터 기반)
```

### 일반 RAG의 작동 방식

```
1. 문서를 작은 조각(chunk)으로 자른다
2. 각 조각을 벡터(숫자 배열)로 변환한다 (임베딩)
3. 질문이 들어오면, 질문도 벡터로 변환한다
4. 질문 벡터와 가장 비슷한 문서 조각을 찾는다 (벡터 유사도 검색)
5. 찾은 조각들을 LLM에게 "참고자료"로 넘기고 답변을 생성한다
```

이것이 **"naive RAG"** 또는 **"vanilla RAG"** 입니다.

### naive RAG의 한계

| 잘 하는 것 | 못 하는 것 |
|-----------|-----------|
| "X가 뭐야?" 같은 단순 검색 | "A와 B의 관계는?" 같은 관계 추론 |
| 한 문서 안의 정보 | 여러 문서에 흩어진 정보 연결 |
| 키워드가 비슷한 문서 찾기 | 의미는 관련있지만 단어가 다른 문서 찾기 |

예시: "ETCH 설비에서 수율이 떨어질 때 어떤 테이블을 봐야 하나?"
- naive RAG: "ETCH"가 포함된 문서 조각만 찾음
- 실제로 필요한 것: ETCH → 챔버 → 센서데이터 → LOT_HISTORY → 통계검정 이라는 **연결고리**

## 2. LightRAG는 일반 RAG와 뭐가 다른가

### 핵심 차이: 지식 그래프(Knowledge Graph)

LightRAG는 문서에서 **엔티티(개체)**와 **관계**를 추출해서 그래프로 만듭니다.

```
[일반 RAG]
문서 → 조각들 → 벡터 DB → 유사도 검색

[LightRAG]
문서 → 조각들 → 벡터 DB → 유사도 검색        ← 이것도 하고
     → 엔티티/관계 추출 → 지식 그래프 → 그래프 검색  ← 이것도 한다
```

### 지식 그래프란?

```
예시: "LOT_HISTORY 테이블은 LOT_ID 컬럼으로 EQP_SENSOR_DATA와 조인된다"

이 문장에서 LightRAG가 추출하는 것:
- 엔티티: [LOT_HISTORY], [EQP_SENSOR_DATA], [LOT_ID]
- 관계: LOT_HISTORY --[조인]--> EQP_SENSOR_DATA (via LOT_ID)
```

이렇게 만들어진 그래프:

```
┌──────────────┐    LOT_ID로 조인    ┌──────────────────┐
│ LOT_HISTORY  │ ──────────────────→ │ EQP_SENSOR_DATA  │
└──────────────┘                     └──────────────────┘
       │                                      │
       │ 포함                                  │ 포함
       ↓                                      ↓
  [LOT_ID, EQP_ID,                      [SENSOR_VALUE,
   STEP_ID, ...]                         TIMESTAMP, ...]
```

### 5가지 쿼리 모드

| 모드 | 무엇을 검색하나 | 비유 |
|------|---------------|------|
| **naive** | 벡터 유사도만 | Google 검색 |
| **local** | 질문에 언급된 엔티티의 직접 이웃 | 위키피디아에서 해당 항목만 보기 |
| **global** | 그래프 전체의 커뮤니티/테마 요약 | 교과서의 "개요" 장 읽기 |
| **hybrid** | local + global | 항목도 보고 개요도 보기 |
| **mix** | hybrid + 벡터 검색 | 가능한 모든 것 동원 |

## 3. LightRAG가 내부적으로 하는 일

### 인덱싱 단계 (문서 투입 시)

```
sample_doc.txt
     │
     ▼
[1단계] 텍스트를 chunk로 분할
     │
     ├──→ [2단계] 각 chunk를 벡터로 변환 (임베딩 API 호출)
     │         → vdb_chunks.json에 저장
     │
     └──→ [3단계] LLM에게 각 chunk를 보내서 엔티티/관계 추출 요청  ← 여기서 비용 발생
               │
               ├──→ 엔티티 → 벡터로 변환 → vdb_entities.json
               ├──→ 관계 → 벡터로 변환 → vdb_relationships.json
               └──→ 그래프 구축 → graph_chunk_entity_relation.graphml
```

**핵심 포인트:**
- 3단계에서 LLM을 많이 호출합니다 (비용 발생)
- LLM이 엔티티/관계를 잘 추출하려면 **입력 문서가 잘 작성**되어야 합니다
- 문서가 모호하면 → 추출되는 엔티티/관계도 모호 → 검색 품질 저하

### 쿼리 단계 (질문 시)

```
"수율 특이점 원인 설비를 찾는 프로세스는?"
     │
     ▼
[1단계] 질문에서 키워드 추출 (LLM 호출)
     │    → "수율 특이점", "원인 설비", "분석 프로세스"
     │
     ├──→ [2단계-a] 벡터 검색: 키워드와 유사한 chunk/엔티티/관계 검색
     │
     └──→ [2단계-b] 그래프 검색: 키워드에 해당하는 엔티티 → 이웃 탐색
               │
               ▼
         [3단계] 검색된 정보를 모아서 LLM에게 전달 → 최종 답변 생성
```

## 4. 당신의 프로젝트에 LightRAG가 맞는가 — 솔직한 의견

### 먼저 당신의 프로젝트를 정리하면

```
[입력]
- DB 테이블들 (LOT_HISTORY, EQP_SENSOR_DATA 등)
- 실시간 수율/센서 데이터

[원하는 것]
- 데이터를 가공해서 통계적 특이점 발견
- 어떤 설비/챔버가 문제인지 분석
- 엔지니어가 판단할 수 있도록 정보 제공

[분석 과정]
테이블 조인 → 데이터 가공 → 통계 검정 → 특이점 식별 → 원인 설비 추적
```

### 솔직한 평가

**LightRAG가 잘 할 수 있는 부분:**

- "ETCH 설비에서 수율이 떨어지면 어떤 분석을 하지?" → 분석 절차 문서를 검색해서 답변
- "LOT_HISTORY와 EQP_SENSOR_DATA를 어떻게 조인하지?" → 테이블 관계 문서에서 답변
- "Commonality Analysis가 뭐야?" → 방법론 설명 문서에서 답변

이것들은 모두 **"지식 검색"** 입니다. 이미 문서에 적혀있는 지식을 찾아서 알려주는 것.

**LightRAG가 할 수 없는 부분:**

- 실제 DB에서 데이터를 꺼내서 계산하는 것
- 통계 검정(t-test, chi-square 등)을 실행하는 것
- 수율 데이터를 실시간으로 모니터링하는 것
- SQL을 실행해서 결과를 가져오는 것

이것들은 **"데이터 처리/분석"** 입니다. LightRAG는 이것을 못합니다.

### 비유로 설명

```
LightRAG = 똑똑한 사서(司書)
- 책(문서)에서 정보를 잘 찾아줌
- "이 분석은 이렇게 하는 거야"라고 설명해줌
- 하지만 직접 실험실에 가서 실험을 하지는 않음

하드코딩 자동화 = 실험실 자동화 장비
- 정해진 절차대로 데이터를 처리하고 분석함
- 직접 SQL을 돌리고, 통계를 계산하고, 결과를 냄
- 하지만 "왜 이렇게 하는 거야?"에 대해 설명은 못함
```

### 결론: 둘 다 필요하지만 역할이 다르다

```
┌─────────────────────────────────────────────────────────┐
│                    당신의 프로젝트                         │
│                                                         │
│  ┌──────────────────┐    ┌──────────────────────────┐   │
│  │  LightRAG 영역    │    │  하드코딩 자동화 영역       │   │
│  │                  │    │                          │   │
│  │ "어떻게 분석하지?" │    │  실제 데이터 분석 실행      │   │
│  │ "무슨 테이블 쓰지?"│    │  SQL 실행, 통계 계산       │   │
│  │ "이 패턴 뭐지?"   │    │  이상치 탐지, 알림         │   │
│  │                  │    │                          │   │
│  │  → 지식/절차 검색  │    │  → 데이터 처리/계산        │   │
│  └──────────────────┘    └──────────────────────────┘   │
│                                                         │
│  둘을 합치면 → "분석 절차를 알고 있고, 실행도 하는 시스템"    │
└─────────────────────────────────────────────────────────┘
```

## 5. 하드코딩 vs LightRAG vs 하이브리드 비교

### 구체적인 시나리오로 비교

**시나리오: "ETCH 3호기 챔버B에서 수율이 3시그마 이상 떨어졌다"**

#### 방법 1: 하드코딩만

```python
# 이미 정해진 로직
def analyze_yield_drop(eqp_id, chamber_id):
    # 1) LOT_HISTORY에서 해당 설비 데이터 조회
    lots = query("SELECT * FROM LOT_HISTORY WHERE EQP_ID=? AND CHAMBER=?", eqp_id, chamber_id)
    
    # 2) 센서 데이터 조인
    sensors = query("SELECT * FROM EQP_SENSOR_DATA WHERE ...")
    
    # 3) t-test 수행
    t_stat, p_value = scipy.stats.ttest_ind(good_lots, bad_lots)
    
    # 4) 결과 리포트
    return {"원인_후보": [...], "p_value": p_value}
```

**장점:** 빠르고, 정확하고, 결과가 일관됨
**단점:** 새로운 분석 패턴이 생기면 코드를 수정해야 함. 분석 이유를 설명 못함.

#### 방법 2: LightRAG만

```python
# 문서에 분석 절차가 적혀있다고 가정
answer = rag.query("ETCH 3호기 챔버B에서 수율이 떨어졌는데 어떻게 분석하지?")
# → "LOT_HISTORY와 EQP_SENSOR_DATA를 조인하고 t-test를 수행하세요..."
```

**장점:** 유연함. 문서만 추가하면 새로운 분석 패턴도 대응 가능.
**단점:** 실제로 SQL을 돌리거나 통계를 계산하지는 않음. "방법"만 알려줌.

#### 방법 3: 하이브리드 (추천)

```
엔지니어: "ETCH 3호기 챔버B 수율 이상"
    │
    ▼
[LightRAG] 분석 절차 검색
    → "이 경우 Commonality Analysis + t-test가 적합합니다"
    → "LOT_HISTORY, EQP_SENSOR_DATA, RECIPE_PARAM 테이블 필요"
    │
    ▼
[자동화 코드] 실제 분석 실행
    → SQL로 데이터 조회
    → 통계 검정 수행
    → 시각화 생성
    │
    ▼
[LightRAG] 결과 해석 지원
    → "p-value 0.002는 유의미합니다. 과거 유사 사례에서는..."
```

### 비교 정리표

| 기준 | 하드코딩 자동화 | LightRAG | 하이브리드 |
|------|---------------|----------|-----------|
| 실제 데이터 분석 | O | X | O |
| 유연한 질의응답 | X | O | O |
| 새 분석패턴 추가 | 코드 수정 필요 | 문서 추가만 | 문서 + 최소 코드 |
| 분석 이유 설명 | X | O | O |
| 정확도/일관성 | 높음 | LLM에 의존 | 높음 |
| 구현 난이도 | 중간 | 낮음 | 높음 |
| 실행 비용 | 서버 비용만 | LLM API 비용 | 둘 다 |
| 오프라인 가능 | O | 로컬 LLM 필요 | 로컬 LLM 필요 |

## 6. 추천하는 접근 방향

### 현실적인 단계별 접근

당장 하이브리드 시스템을 만들기보다, 단계적으로 접근하는 것을 추천합니다.

#### 1단계: 지금 하고 있는 것 (LightRAG 학습)

지금 하고 있는 것이 맞습니다. LightRAG에 도메인 문서를 넣어보면서:
- 지식 그래프가 어떻게 만들어지는지 확인
- 어떤 형태의 문서를 넣어야 좋은 그래프가 나오는지 실험
- 5가지 쿼리 모드의 차이를 체감

**이 단계의 목표: "LightRAG가 우리 도메인 지식을 잘 이해하는가?"를 판단하는 것**

#### 2단계: 문서 품질 개선

LightRAG의 성능은 **입력 문서의 품질**에 거의 100% 달려있습니다.

좋은 문서 예시:
```
# 수율 특이점 분석 프로세스

## 1단계: 데이터 수집
LOT_HISTORY 테이블에서 최근 7일간의 LOT 데이터를 조회한다.
조건: STEP_ID = 'ETCH', YIELD < (평균 - 3*표준편차)

## 2단계: 설비 매칭
LOT_HISTORY.EQP_ID로 EQP_SENSOR_DATA를 조인한다.
조인키: LOT_ID, STEP_SEQ

## 3단계: 통계 검정
정상 LOT과 불량 LOT의 센서값을 Welch t-test로 비교한다.
p-value < 0.05인 센서를 원인 후보로 선정한다.
```

나쁜 문서 예시:
```
수율이 떨어지면 여러 테이블을 보고 분석합니다.
통계적 방법을 사용하여 원인을 찾습니다.
```

→ 구체적인 테이블명, 컬럼명, 조인 조건, 통계 방법이 명시될수록 좋습니다.

#### 3단계: 판단 — LightRAG를 계속 쓸지 결정

2단계까지 해보면 답이 나옵니다:

**LightRAG를 계속 쓰는 게 맞는 경우:**
- 분석 절차가 다양하고 자주 바뀜
- 엔지니어마다 다른 방법론을 사용
- "왜 이 분석을 하는가"에 대한 설명이 중요
- 신입 엔지니어 교육/온보딩 용도

**하드코딩이 더 나은 경우:**
- 분석 절차가 5~10개로 정해져 있고 거의 안 바뀜
- 정확하고 일관된 자동 실행이 핵심
- LLM API 비용이 부담
- 폐쇄망에서 외부 LLM 호출이 어려움

**하이브리드로 가는 경우:**
- 위의 두 가지가 다 필요
- 장기적으로 시스템을 고도화할 계획

#### 4단계 (하이브리드로 갈 경우): 아키텍처

```
┌────────────────────────────────────────────┐
│             엔지니어 인터페이스               │
│         (웹 UI 또는 챗봇 형태)               │
└──────────────────┬─────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────┐
│          오케스트레이터 (LLM Agent)     │
│                                      │
│  1. 사용자 질문 이해                    │
│  2. LightRAG에서 분석 절차 검색         │
│  3. 필요한 도구(SQL, 통계) 선택         │
│  4. 도구 실행 및 결과 종합              │
│  5. 엔지니어에게 결과 + 해석 제공        │
└───┬──────────────┬───────────────┬───┘
    │              │               │
    ▼              ▼               ▼
┌────────┐  ┌───────────┐  ┌───────────┐
│LightRAG│  │ SQL 실행기 │  │ 통계 엔진  │
│지식 검색│  │ DB 조회    │  │ scipy 등  │
└────────┘  └───────────┘  └───────────┘
```

이것은 **LLM Agent** 패턴이며, LightRAG는 그 안에서 "지식 검색 도구" 역할을 합니다.
이 방향으로 갈 경우 LangChain, LlamaIndex 같은 프레임워크도 함께 검토하게 됩니다.

## 부록: 자주 오해하는 것들

### "RAG를 쓰면 LLM이 우리 데이터를 학습하는 거 아닌가요?"

아닙니다. RAG는 LLM을 재학습(fine-tuning)시키는 것이 아닙니다.
매번 질문할 때마다 관련 문서를 **검색해서 프롬프트에 붙여주는 것**입니다.
LLM 자체는 변하지 않습니다.

### "지식 그래프가 자동으로 완벽하게 만들어지나요?"

아닙니다. LLM이 추출하는 것이라 오류가 있을 수 있습니다.
- 엔티티를 놓치거나
- 관계를 잘못 해석하거나
- 같은 것을 다른 이름으로 추출하거나 (LOT_HISTORY vs lot_history)

문서가 명확할수록 추출 품질이 올라갑니다.

### "LightRAG가 SQL을 대신 짜주나요?"

아닙니다. LightRAG는 "이 경우 이런 SQL을 쓰라"는 **문서에 적힌 내용**을 찾아줄 뿐입니다.
실제 SQL을 생성하고 실행하려면 별도의 Text-to-SQL 도구가 필요합니다.

### "폐쇄망에서 LightRAG를 쓸 수 있나요?"

가능하지만 조건이 있습니다:
- **인덱싱**: LLM API 호출 필요 (사내 LLM API로 대체 가능 — 이미 설정함)
- **임베딩**: 임베딩 API 필요 (사내 임베딩 서버 또는 로컬 모델)
- **쿼리**: LLM API 호출 필요

사내 LLM API가 있으니 가능은 하지만, **LLM의 한국어 성능과 엔티티 추출 능력**이
GPT-4o-mini보다 떨어질 수 있습니다. 이것은 실제로 테스트해봐야 알 수 있습니다.
